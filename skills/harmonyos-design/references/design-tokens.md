# Design Token 与视觉约束

`HarmonyOSComponentUXExamples` 把视觉常量拆为独立 token 文件，这比把视觉值直接写在页面里更适合长期维护。

## 推荐拆分

- `Color.ets`：品牌强调色、弱化色、补充色。
- `Padding.ets`：内边距。
- `Space.ets`：组件间距。
- `CornerRadius.ets`：圆角。
- `Size.ets`：固定尺寸、百分比宽高、少量特殊尺寸。

## 使用原则

- 语义优先，不要只按数值命名。
- 如果项目早期必须按数值命名，后续至少在业务稳定后抽出语义别名。
- 同一类尺寸不要同时出现多个非常接近的值，例如 `15`、`16`、`17` 混用。
- 组件级 token 复用页面级 token，不要平行维护两套体系。

## 从示例仓库提炼出的视觉规律

- 页面与列表常用间距集中在 `4vp`、`8vp`、`12vp`、`16vp`、`24vp`、`32vp`。
- 强调色示例为 `#0A59F7`，但业务项目应改为你自己的品牌语义色，而不是直接照搬。
- 列表项、卡片、点击区域会结合圆角、点击态背景和系统色资源一起使用。
- 百分比宽高用于页面骨架和导航宽度，固定尺寸用于控件、图标、弹层和局部布局。

## 建议命名

优先使用语义名：

- `COLOR_BRAND_PRIMARY`
- `COLOR_TEXT_SECONDARY`
- `SPACE_PAGE_HORIZONTAL`
- `SPACE_SECTION_GAP`
- `RADIUS_CARD`
- `SIZE_TOUCH_TARGET_MIN`

仅当仓库已有严格的数值常量风格时，再保留数值常量并补一个语义层。

## 资源使用建议

- 能使用系统资源色时优先使用 `$r('sys.color.xxx')`。
- 能使用系统字体与图标资源时优先使用系统资源。
- 自定义色只放在品牌强调、业务状态、特殊插画等系统资源无法覆盖的地方。

## 代码组织建议

- 不要在业务组件里直接拼大段颜色和尺寸字面量。
- 页面样式如果被多个组件共享，提炼为 token 或 style builder。
- 对按钮、标题栏、列表项、表单输入框等高复用组件，建立统一尺寸和状态规范。

## 评审清单

- 页面中是否仍有散落的硬编码颜色和尺寸。
- 主按钮、次按钮、禁用按钮是否符合统一层级。
- 卡片、列表、弹层的圆角和间距是否一致。
- 同一状态色是否在不同页面出现不一致。

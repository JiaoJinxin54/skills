# HarmonyOS Design 实施清单

把业务需求转成 ArkUI 设计方案时，按下面顺序执行。

## 1. 先定义页面角色

- 首页、列表页、详情页、设置页、表单页、弹层页、浮层反馈页。
- 明确用户进入页面后最先要完成的动作。

## 2. 决定信息架构

- 顶层目的地：`BottomTabBar`
- 页面内分组：`SubTab`
- 页头识别与顶部动作：`TitleBar`
- 一次性确认或轻操作：`Dialog` / `BindSheet`

如果信息架构不稳定，不要先进入视觉细节。

## 3. 选择交互组件

- 主提交：`Button`
- 补充操作：`Menu`
- 高频一组操作：`ActionBar`
- 搜索：`Search`
- 文本输入：`TextInput` / `TextArea`
- 数量调节：`Counter`
- 单项选择：`Radio`
- 开关状态：`Toggle`
- 连续值：`Slider`

## 4. 设计反馈链路

- 成功或轻反馈：`Toast`
- 可撤销或带动作反馈：`SnackBar`
- 耗时过程：`Progress` / `LoadingProgress`
- 状态提醒：`Badge`

不要只设计成功态，必须同时设计错误态、空态和加载态。

## 5. 抽离公共层

- Token：颜色、尺寸、圆角、间距。
- Builder：可复用视图片段。
- Model：页面状态、断点、列表项数据。
- Utils：窗口、日志、防抖、公用格式化。

## 6. 输出给代码时要包含的内容

- 页面结构草图或组件层级。
- 组件选型理由。
- 关键状态和交互流转。
- token 建议。
- 响应式规则。
- 需要抽成通用组件的部分。

## 7. 常见错误

- 把所有状态都堆在一个页面组件里。
- 直接硬编码视觉值，后续无法统一。
- 用自定义容器替代系统组件，丢掉原生感。
- 把 `Toast` 用成确认弹窗。
- 把 `BottomTabBar` 用成局部筛选器。
- 没有考虑键盘避让、安全区和横屏。

## 8. 交付模板

可以按下面的格式输出设计方案：

1. 页面目标与主任务
2. 信息架构与导航方式
3. 组件选型
4. 状态设计
5. token 建议
6. 响应式与设备适配
7. ArkTS 落地建议
